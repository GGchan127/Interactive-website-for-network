// Introduction

STANDARDS-BASED MODELS

Standards-based models are created so that network devices and their operations follow common rules, allowing devices from different manufacturers to work together seamlessly. The goal is to reduce proprietary dependencies as much as possible.

Two major standards bodies define the reference models that the entire networking industry follows today:

IEEE Institute of Electrical & Electronics Engineers

Defined the Open Systems Interconnection (OSI) Model — a 7-layer reference model that helps us understand what layer each device operates at and gives us a "big-picture" view of how networks function. Ideal for learning and troubleshooting.

IETF Internet Engineering Task Force

Defined the TCP/IP Model (Transmission Control Protocol / Internet Protocol) — 4 to 5 layers, used as the actual protocol model running on the real Internet today. Their specifications are published as RFCs (Request for Comments).

🎯 Theory vs Reality

OSI Model is the teaching standard — a clean 7-layer framework perfect for learning concepts and diagnosing problems layer by layer.

TCP/IP Model is the working standard — what every real-world Internet device actually uses to communicate.

📘 What is a Protocol?

A protocol is a set of rules, agreements, or "language" used for data communication between computers or network devices.

// The Problem

WHY DO WE NEED STANDARDS?

Without standards-based models, every vendor would build their own incompatible systems. Here's what that world looks like — and why standards save us from it.

🔒 Proprietary World BAD

If there were no standards-based models, the problems we'd constantly face are:

  • Devices from different brands cannot work together
  • You must buy everything from one vendor — routers, switches, software, all of it
  • Applications & software locked to a single manufacturer
  • Vendor lock-in means high prices and zero flexibility
  • Users always lose — fewer choices, more cost

🔓 Standards-Based World GOOD

With open standards, the networking industry transforms:

  • Cisco router can talk to Juniper router, no problem
  • Mix & match equipment from any vendor
  • Software runs on any compatible hardware
  • Competitive market drives innovation and lowers prices
  • Users win — freedom of choice and interoperability

Real-World Scenario: Proprietary Failure

Imagine two routers connected on a link, but they're proprietary. If the right-side router fails, you must buy the exact same brand to restore the connection. With standards-based devices, you could swap in any vendor and everything keeps working.

HOW DATA TRAVELS

When data travels from source to destination computer, here's the step-by-step journey across the network layers:

Fig 3.1 — Encapsulation across the OSI 7 layers: each layer adds its header to the DATA.

01
Check Source & Destination IP (Layer 3)

The source computer checks whether the source and destination IPs are on the same network/subnet. If they are, data is sent directly to the destination IP. If not, data is sent to the computer's configured Gateway.

Layer 3IP AddressSubnet Check
02
ARP Lookup & Frame Encapsulation (Layer 2)

If sending to the Gateway first, the computer checks the ARP Table (ARP Cache) for the Gateway's MAC Address. Then it performs encapsulation by wrapping the data with the MAC Address — a new Frame is born.

Layer 2ARP TableMAC AddressEncapsulation
03
Switch Forwards the Frame (Layer 2)

The Frame is forwarded to a Switch. The Switch looks up the MAC Address to decide which interface to send the Frame out on. The Frame travels onward to the Gateway — a Layer 3 device (Router, Layer 3 Switch, or Firewall).

SwitchMAC LookupFrame Forwarding
04
Router Processes & Re-Encapsulates (Layer 3)

When the Router receives the Frame, it verifies the Checksum, performs de-encapsulation to strip off the old MAC Address, then rebuilds a new Frame with new source/destination MACs (Layer 2). The Layer 3 Packet is also rebuilt with TTL decreased by 1 and a new Checksum calculated before forwarding.

RouterDe-EncapsulationTTL−1New Checksum
05
Arrival at Destination

When data reaches the destination computer, it undergoes de-encapsulation to remove the MAC Address and IP Address, then passes upward through each higher layer until the original data sent by the source computer is finally visible to us.

DestinationDe-EncapsulationLayer 7 Delivery

// The Reference Model

OSI 7 LAYER MODEL

The OSI Model (Open Systems Interconnection) is divided into 7 layers. Click any layer to see what protocols and devices operate at that level.

7
Application PDU: Data
Interface between application software (e.g. Web Browser) and the network protocol.
Defines how applications running on a computer communicate with software on the system. For example, an HTTP application must be called by a Web Browser such as Firefox, Microsoft Edge, or Chrome.
HTTP / HTTPSFTPSMTPDNSWeb Browser
6
Presentation PDU: Data
Translates, encrypts, and compresses data so applications can read it correctly.
Handles data formatting, encryption/decryption, and compression. The "translator" of the stack.
SSL / TLSJPEG / PNGASCIIEncryption
5
Session PDU: Data
Establishes, manages, and terminates communication sessions between devices.
Keeps conversations organized — opens connections, maintains them, and closes them cleanly.
NetBIOSRPCSession tokens
4
Transport PDU: Segment
Deals primarily with TCP / UDP protocols — closely tied to the Application Layer.
Breaks data into segments and ensures reliable (TCP) or fast (UDP) delivery between hosts. Uses port numbers to identify applications.
TCPUDPPort numbers
3
Network PDU: Packet
Standards for Routing protocols, IPv4, and IPv6. Devices include Routers, L3 Switches (Multilayer Switches).
Adds IP addresses and decides the best path across networks. This is where Routers operate.
IPv4 / IPv6Routing ProtocolsRouterL3 Switch
2
Data Link PDU: Frame
Standards for encapsulation: Ethernet, HDLC, PPP, MAC Address standards. Devices: Switch, Bridge, Access Point.
Packages bits into frames and uses MAC addresses for local delivery. This is where Switches operate.
EthernetHDLCPPPMAC AddressSwitchBridgeAccess Point
1
Physical PDU: Bits
Standards for Media Types: Twisted Pair, Fiber Optic, HUB, Repeater, Connector Types. Covers digital signals — bits 1 and 0.
The actual physical medium that carries raw bits as electrical, light, or radio signals. The PDU at this layer is called a Bit.
Twisted PairFiber OpticHUBRepeaterConnectorsVoltage / Bits
Memory trick (top → bottom): "All People Seem To Need Data Processing" → Application, Presentation, Session, Transport, Network, Data Link, Physical.

// The Working Model

TCP/IP MODEL

The TCP/IP Model has 5 layers, defined by the IETF, used as the protocol model for the actual Internet. When compared with the OSI Model, the two correspond as shown below.

TCP/IP 5-Layer Architecture

Fig 2.2 — The 5-layer TCP/IP model defined by IETF.

OSI vs TCP/IP — Side by Side

OSI Model

7 layers · Reference standard by IEEE

7 Application
6 Presentation
5 Session
4 Transport
3 Network
2 Data Link
1 Physical

TCP/IP Model

5 layers · Defined by IETF · The real Internet

Application = OSI layers 7 + 6 + 5 HTTP, FTP, DNS, SMTP
Transport = OSI 4 · TCP, UDP
Internet = OSI 3 · IP, routing
Data Link = OSI 2 · Ethernet, MAC
Physical = OSI 1 · Cables, Wi-Fi
Key takeaway: The OSI model is the theory (great for understanding and troubleshooting layer-by-layer). TCP/IP is the practice (what actually runs on every device on the Internet).
