// Internet Routing
What is BGP and What is Path-Vector?

BGP (Border Gateway Protocol) is the standard routing protocol used to forward data packets across the Internet. It is an Exterior Gateway Protocol (EGP) — meaning it operates between different organisations, unlike OSPF or EIGRP which run within a single organisation (IGP).

Think of BGP as the "postal system of the Internet" — when sending data, BGP finds all possible paths and selects the best one, which typically means hopping across multiple Autonomous Systems.

The 3 Types of Routing Protocols

Distance-Vector RIP · EIGRP · Used in small LANs

Imagine standing at a crossroads with a sign that reads "Bangkok — 200 km". The sign tells you the distance, but nothing about which provinces you pass through, where to turn, or whether there is a motorway.

A Distance-Vector router works the same way. It only knows: "destination 10.0.0.0 is 3 hops away — send out this interface." It has no idea which routers the packet passes through along the way.

Distance-Vector routers exchange information simply by telling their neighbours: "I can reach that destination with cost X." The neighbour adds 1 and passes it on — like a rumour spreading by word of mouth. The problem is that if a link fails, incorrect information can loop back and forth for a long time before everyone learns the truth (Routing Loop).

Link-State OSPF · IS-IS · Used in enterprise / data centres

Instead of a roadside sign, imagine you have a GPS with the complete map of every road in the country — every road length, traffic level, and current closure. You can compute the best route instantly, and if a road closes the GPS recalculates immediately without waiting for anyone else.

A Link-State router works this way. Each router announces its own links to the entire network: "I am connected to these neighbours with these costs." Every router collects that data, builds an identical map, then each independently runs Dijkstra's Algorithm to compute the shortest path to every destination.

The result is that all routers share the same view of the network and respond to changes very quickly — ideal for medium-to-large single-organisation networks.

Path-Vector BGP only · Used on the Internet / between ASes

Both Distance-Vector and Link-State share the same goal: "find the shortest path." But on the real Internet, the shortest path is not always the most important consideration.

Suppose you are an ISP with two paths to a destination. Path A is shorter but passes through a competitor's network — very expensive transit. Path B is slightly longer but uses a partner with a free peering agreement. Which do you choose? Path B, even though it is longer.

Distance-Vector and Link-State cannot make that decision — they do not even know "who" the path passes through. BGP, as a Path-Vector protocol, always attaches the full list of every AS on the route. For example: "network 8.8.8.0/24 reachable via AS200 → AS300 → AS400." This is called the AS_PATH, and it enables two things no other protocol can do:

First — automatic loop prevention. If a router sees its own AS in the AS_PATH, the route has looped back and is discarded immediately without any special mechanism.

Second — policy-based decisions. An admin can say "never select paths through a competitor's AS" or "prefer this AS as entry point for that traffic" — exactly what ISPs need because their business relationships are complex, not simply "take the shortest path."

AS_PATH — BGP knows the complete path across Autonomous Systems

// Key Features
BGP Purpose and Key Features

BGP makes peering possible — the agreement between two networks to exchange routing information. Without BGP, networks in different organisations could never reach each other across the Internet.

Why does this sound contradictory? The confusion comes from "network" having multiple meanings. Within the same AS (e.g. inside AIS) BGP is not needed — OSPF is enough to route normally. But between different ASes, such as a computer on AIS accessing a website on AWS, there is no path without BGP because nothing tells the router "AWS is in this direction."

7 Key Features

Inter-AS Connectivity
The heart of BGP — two ASes (e.g. TRUE and Cloudflare) agree to peer and exchange routing information. Both sides must configure matching parameters before they can communicate.
Next-Hop Support
When BGP announces "network 8.8.8.0/24 is reachable," it must also specify which IP to send the packet to first (next-hop). For eBGP the next-hop is usually the peer router's IP. For iBGP the next-hop remains the original eBGP router's IP (use next-hop-self when needed).
BGP Speaker Coordination
Large ASes contain multiple BGP routers (BGP speakers). They must talk to each other via iBGP to keep a consistent routing table inside the AS — otherwise one router sends traffic left while another sends it right.
Path Information
Every BGP route update carries an AS_PATH — e.g. "network 1.1.1.0/24 came via AS200→AS300→AS400." This serves two purposes: (1) loop prevention — if the router's own AS is in the path, the route is discarded; and (2) best-path comparison.
Policy Support
BGP lets admins define: "accept this route? advertise that route? prefer this path over that one?" via route-maps, prefix-lists, communities. This is why large ISPs use BGP — they need precise traffic control, not just "shortest path."
Runs over TCP
BGP uses TCP Port 179 as its transport — no need to build its own reliability mechanism. TCP guarantees that updates arrive in full and in order, unlike OSPF/EIGRP which use multicast directly.
CIDR & MD5 Security
BGP supports Classless Inter-Domain Routing (CIDR) — e.g. 10.0.0.0/22 can be advertised instead of 4 separate networks. It also supports MD5 authentication between peers to prevent BGP hijacking.

What is an Autonomous System (AS)?

An AS (Autonomous System) is a group of routers and networks under the control of a single organisation that shares a common routing policy. Each AS receives a unique ASN (AS Number) assigned by IANA/APNIC.

Organisations that hold an AS include ISPs, cloud providers, CDNs, universities, and large companies connected to multiple Internet providers (multi-homed).

// Session Types
eBGP vs iBGP

BGP has two forms depending on whether the session is within the same AS or between different ASes.

eBGP between AS 100 and AS 200 — iBGP within each AS
eBGP — External BGP
Between different ASes. Administrative Distance = 20 (high trust). Peers must be directly connected (or use ebgp-multihop). AS_PATH is extended every time the route crosses an AS. Think of it as "international diplomacy."
iBGP — Internal BGP
Within the same AS. Administrative Distance = 200 (lower trust than IGP). Used because large ASes have multiple BGP routers that all need to share eBGP-learned routes. Key rule: routes from an iBGP peer are not forwarded to another iBGP peer (loop prevention).
When is iBGP required? A small AS with only one BGP router does not need iBGP at all — a single eBGP session to the outside world is sufficient. iBGP only becomes necessary when an AS has multiple BGP routers that need to share the same routes internally.

// Session States
BGP Finite State Machine — 6 States

The FSM (Finite State Machine) is the internal mechanism that runs every time a BGP router tries to establish a neighbour relationship. Every time you enter neighbor x.x.x.x remote-as YYY on Cisco, the FSM starts immediately.

1. Idle
Just booted / BGP reset
BGP is inactive. It rejects all incoming connections and waits for a start event.
Analogy: A phone that is switched off — no calls can get through.
2. Connect
Initiates TCP connect to Port 179
Waiting for TCP 3-way handshake (SYN → SYN-ACK → ACK). Success → OpenSent. Failure → Active.
Analogy: Dialling a friend and waiting for them to pick up.
3. Active
TCP failed — retrying
Repeatedly attempting TCP connection. Stuck in Active for a long time usually means a firewall is blocking port 179 or the neighbour IP is wrong.
Analogy: Calling over and over but nobody answers.
4. OpenSent
TCP established — OPEN message sent
OPEN contains: BGP version (4), AS number, Hold Time, Router ID. Waiting for OPEN from the peer. Any error → NOTIFICATION sent → back to Idle.
Analogy: Both sides shaking hands and introducing themselves.
5. OpenConfirm
OPEN accepted — waiting for KEEPALIVE
Waiting for a KEEPALIVE to confirm the session. Received → Established. Hold Timer expires first → back to Idle.
Analogy: After introductions, waiting for a confirming nod: "OK, we're neighbours."
6. Established
KEEPALIVE received — Session UP!
The goal state. UPDATE messages exchange routes. KEEPALIVE sent every 60 seconds. If Hold Timer (180 s) expires → back to Idle.
Analogy: Now friends — start sharing routes freely.
Troubleshooting with State Run show ip bgp summary and check the State/PfxRcd column. If you see Active — stuck at state 3 (TCP unreachable — check firewall / IP). If you see a number such as 1 — session is Established and 1 route has been learned.
06
// Protocol Messages
BGP Message Types
OPEN
Sent after TCP is established to initiate the BGP session. Contains: BGP version, AS number, Hold Time, BGP Identifier (Router ID).
KEEPALIVE
Sent every 60 seconds (default) to confirm the session is still alive. If no KEEPALIVE is received within the Hold Timer (180 s) — the session is torn down.
UPDATE
Carries routing information — announces new networks (Advertise) or withdraws existing ones (Withdraw). Always includes Path Attributes.
NOTIFICATION
Sent when an error occurs — reports the reason and immediately closes the session. The router resets to Idle. Examples: "AS number mismatch", "Hold Timer expired."

// Path Selection
BGP Attributes and Best Path Selection

When BGP knows multiple paths to the same destination, it applies the Best Path Selection Algorithm — comparing attributes one by one in order. If two routes tie on one attribute, the comparison moves to the next.

Selection Order — W·L·L·A·O·M·E·I Weight → Local Preference → Locally Originated → AS_PATH → Origin → MED → EBGP over iBGP → IGP Metric
1 Weight Higher is better · default 32768 (local) / 0 (learned) Cisco only

Weight is a Cisco-proprietary attribute — it is not part of the BGP standard and therefore exists only on Cisco routers. Critically, it is never advertised to any neighbour; it only affects the router where it is configured.

Use case: Force a single router to prefer one upstream ISP over another without affecting any other router in the AS. Set a higher Weight for routes from the preferred ISP.

Configuration: neighbor x.x.x.x weight 500 — applies to all routes from that neighbour. Use a route-map to apply it selectively per prefix.

Verify: show ip bgp x.x.x.x (detailed) shows the Weight of each path.

2 Local Preference Higher is better · default 100 Within same AS

Local Preference tells every router in the AS: "which exit point should we use to leave our AS?" Higher value = preferred. It is shared among all BGP routers within the AS via iBGP but is never sent to other ASes.

Use case: Your AS connects to two ISPs — ISP-A and ISP-B. You want all traffic to exit via ISP-A regardless of which internal router originates it. Set Local Preference for ISP-A routes to 200 and leave ISP-B at 100. Every router in the AS will unanimously choose ISP-A. This is the key difference from Weight, which only affects one router.

Configuration: Use a route-map with set local-preference 200, then apply it with neighbor x.x.x.x route-map NAME in.

Verify: show ip bgp — column LocPrf.

3 Locally Originated network / aggregate preferred over redistribute Local only

If Weight and Local Preference are equal, BGP checks whether either route was originated by this router itself. Routes injected via the network command or aggregation are preferred over routes that were redistributed from another protocol.

In practice this attribute rarely plays a role because a single router almost never originates the same prefix via two different methods simultaneously.

4 AS_PATH Shorter (fewer ASes) is better Cross-AS

AS_PATH is the ordered list of every AS the route has passed through. For example, a route originating in AS400 arriving at AS100 via AS200 and AS300 has AS_PATH "300 200 100" (newest first). BGP prefers the route with the shorter AS_PATH — fewer ASes traversed.

Loop prevention: If a router sees its own AS number in an incoming AS_PATH, it knows the route has looped back and discards it immediately — automatically, with no extra mechanism needed.

AS_PATH Prepending: A widely used multi-homing technique where you deliberately repeat your own AS number to make a path appear longer, steering inbound traffic from the Internet toward a preferred link. set as-path prepend 65001 65001 65001 adds your AS three times.

Verify: show ip bgp — column Path shows the AS numbers traversed.

5 Origin i (IGP) > e (EGP) > ? (Incomplete) Cross-AS

Origin indicates how the route was originally injected into BGP. There are three possible values:

i (IGP) — Someone used the network command to inject this route directly. Most trustworthy — it means the network owner deliberately announced it.

e (EGP) — Came from the legacy EGP protocol (predecessor of BGP, now obsolete). Rarely seen in modern networks.

? (Incomplete) — Route was redistributed into BGP from another protocol (e.g. OSPF or a static route). Least trusted because BGP does not know the exact original source.

Configuration: Normally not set directly — it is assigned automatically. Using network gives you i; redistribution gives you ?.

Verify: show ip bgp — the last character in the Path column, e.g. 100 200 i means AS_PATH is 100, 200 and Origin is IGP.

6 MED (Multi-Exit Discriminator) Lower is better · default 0 Between ASes

MED is a "suggestion" sent outward to a neighbouring AS: "if you want to send traffic to us, please use this entry point." Lower value = preferred. Default is 0.

Use case: Your AS has two links into AS200 — via R1 and R2. You want AS200 to send inbound traffic through R1. Set MED on R1 to 10 and on R2 to 100. AS200 sees that R1 has the lower MED and prefers it.

Note: MED controls inbound traffic (traffic entering your AS), while Local Preference controls outbound traffic (traffic leaving your AS). MED is only compared between routes from the same neighbouring AS.

Configuration: Route-map with set metric 10, applied outbound: neighbor x.x.x.x route-map NAME out.

Verify: show ip bgp — column Metric.

7 eBGP over iBGP eBGP always preferred over iBGP Fixed rule

This is a hard rule — not a configurable attribute. If the same prefix is learned from both an eBGP peer and an iBGP peer, BGP always selects the eBGP route.

The reason: eBGP routes come directly from outside the AS, so the information is considered more accurate. An iBGP route has already been relayed one hop inside the AS. This is consistent with the Administrative Distance values: eBGP = 20, iBGP = 200.

8 IGP Metric (Lowest cost to Next-Hop) Lower is better Local

When all previous attributes are equal, BGP compares how far the next-hop of each route is from this router in terms of IGP cost (e.g. OSPF cost or EIGRP metric), and prefers the route whose next-hop is closest.

In practice: This is usually an automatic tiebreaker — no manual configuration needed. It matters in an AS with multiple iBGP routers each pointing to different next-hops; BGP picks the one reachable at lowest IGP cost.

Verify: show ip bgp x.x.x.x (detailed) shows "metric from next hop."

Attribute Configuration — The Common Tool Most attributes are set using route-maps — the primary BGP policy tool. A route-map is attached to a neighbour with neighbor x.x.x.x route-map NAME in/out, then uses set commands inside the route-map to assign attribute values and match commands to target specific prefixes.

// Cisco IOS
Cisco Commands and Configuration Examples

Basic Commands

Cisco IOS — BGP Basic
! Start BGP process
router bgp <AS-number>

! Define a neighbour / peer
neighbor <ip-address> remote-as <AS-number>

! Advertise a network into BGP
network <network> mask <subnet-mask>

! Set Router-ID manually
bgp router-id <ip-address>

! iBGP — use loopback as session source
neighbor <ip> update-source loopback 0

eBGP Example — R1 (AS65001) ↔ R2 (AS65002)

eBGP Configuration
! ==== R1 (AS 65001) ====
R1(config)# router bgp 65001
R1(config-router)# bgp router-id 1.1.1.1
R1(config-router)# neighbor 192.168.12.2 remote-as 65002
R1(config-router)# network 10.1.0.0 mask 255.255.0.0
R1(config-router)# no auto-summary

! ==== R2 (AS 65002) ====
R2(config)# router bgp 65002
R2(config-router)# bgp router-id 2.2.2.2
R2(config-router)# neighbor 192.168.12.1 remote-as 65001
R2(config-router)# network 10.2.0.0 mask 255.255.0.0
R2(config-router)# no auto-summary

iBGP Example — Using Loopback

iBGP Configuration
! R1 and R2 are in the same AS (65001)
R1(config)# interface loopback 0
R1(config-if)# ip address 1.1.1.1 255.255.255.255

R1(config)# router bgp 65001
R1(config-router)# neighbor 2.2.2.2 remote-as 65001  ← same AS = iBGP
R1(config-router)# neighbor 2.2.2.2 update-source loopback 0
R1(config-router)# network 10.1.1.0 mask 255.255.255.0
