PROTOCOL

ARP translates a known IP address into a MAC address — bridging Layer 3 (IP) and Layer 2 (Ethernet) so that a Switch can forward frames to the correct physical port.

01
// Slide 3 — Definition

WHAT IS ARP?

When PC 1 wants to send data to 192.168.1.100, it hands the frame to a Switch. The Switch operates at Layer 2 and works only with MAC addresses — not IP. ARP is the protocol that finds the MAC address that belongs to a given IP address, so Layer 2 forwarding can happen.

THE PROBLEM
PC 1 knows the destination IP (e.g. 192.168.1.100) but its frame's Destination MAC field is empty (?????.?????.?????). Without a MAC address the Switch cannot forward the frame.
THE SOLUTION
ARP sends a broadcast — "Who has 192.168.1.100? Tell me your MAC!" — and the owner replies with its MAC. PC 1 can then fill in the Ethernet header and send the actual data.
THE RESULT
The IP↔MAC mapping is stored in the ARP Cache on the PC. Future packets to the same IP skip the broadcast entirely — the Switch just forwards.
💡 REAL-WORLD ANALOGY (from the video)
Nickname (IP address)
Everyone knows you as "Bee" — but the Switch (postal system) can only deliver using your real full name.
Full Name (MAC address)
ARP asks the whole room: "Hey — Bee! What's your full name?" Bee (192.168.1.100) replies with its full name (CCCC.CCCC.CCCC) and the Switch can now deliver the letter.

HOW ARP WORKS

STEP 1 — ARP REQUEST (Broadcast)
  • PC 1 broadcasts to FFFF.FFFF.FFFF (everyone)
  • "Who has IP 192.168.1.100? Tell 192.168.1.40"
  • Switch floods out all ports — every device receives it
  • Only the device with that IP responds
STEP 2 — ARP REPLY (Unicast)
  • PC 2 replies directly to PC 1 (unicast)
  • "My MAC is CCCC.CCCC.CCCC"
  • PC 1 stores this in its ARP Cache
  • Data transfer can now begin with correct L2 header

ARP PACKET STRUCTURE

An ARP frame has two layers: the Ethernet header (for L2 delivery) wrapping the ARP payload (the actual question or answer). Here are the exact fields from the video.

ARP REPLY — PC 2 (192.168.1.100) answering PC 1

Ethernet Header ARP Reply Payload
Dest MAC Src MAC Src MAC Src IP Dest MAC Dest IP
AAAA.AAAA.AAAA CCCC.CCCC.CCCC CCCC.CCCC.CCCC 192.168.1.100 AAAA.AAAA.AAAA 192.168.1.40
Unicast to PC 1 PC 2's MAC (the answer!) PC 2's MAC PC 2's IP PC 1's MAC PC 1's IP
Key difference:   ARP Request = Broadcast (FFFF.FFFF.FFFF) · Destination MAC unknown (0000.0000.0000)  |  ARP Reply = Unicast · Destination MAC is now known (AAAA.AAAA.AAAA)

ARP CACHE

After a successful ARP exchange, the IP↔MAC mapping is stored locally in the ARP Cache (also called the ARP Table). Next time the same destination is needed, no broadcast is needed — saving bandwidth and time.

DYNAMIC ENTRIES

  • Created automatically by ARP exchanges
  • Have a TTL (Time To Live) — typically 2 minutes on Windows
  • Automatically removed when TTL expires
  • Re-acquired on next communication attempt

STATIC ENTRIES

  • Manually added by an administrator
  • Never expire — permanent until deleted
  • Used for critical devices (e.g. gateway, servers)
  • Added with arp -s <IP> <MAC>

COMMAND REFERENCE

arp -a
Display the full ARP cache (all interfaces)
arp -d *
Delete all dynamic ARP cache entries (Windows)
arp -s <IP> <MAC>
Add a static ARP entry manually
arp -d <IP>
Remove a specific entry from the cache
What the Switch learns:   While the PC uses ARP to learn MAC→IP mappings, the Switch builds its own MAC Address Table — it records which port each MAC arrived on (PC → MAC → Switch port). The Switch does NOT need ARP; it learns purely from incoming frames.
